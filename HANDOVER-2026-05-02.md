## Architecture quick-ref (don't re-derive this; it's not in CLAUDE.md)

- **Tool registry** = `electron/llmTools.cjs` `TOOLS` array. Each entry: `{name, description, isWrite, parameters, storageCall}`.
- **Dispatch** = `electron/toolRouter.cjs` two tables (`READ_DISPATCH` / `WRITE_DISPATCH`) keyed by tool name, each is `(storageFn, args) => result`.
- **Sentinels for non-storage tools**: `__zotero__` (HTTP client), `__exporter__` (export formats: markdown/html/json/share use storage funcs, docx/latex are lazy-loaded from their own modules in the dispatch).
- **Approval flow**: `executeToolCall` in `llmClient.cjs` line ~264. `isWrite` tools wait for `llm:approve` IPC unless `autoApproveTools` storage flag is true OR session has `alwaysAllow.has(name)`.
- **MCP server**: `electron/mcp-server.cjs` registers every TOOLS entry with the SDK + 5 resources + 4 prompts. Two entry points:
  - `electron/main.cjs --mcp-server` (Electron-wrapped; required when running from .exe)
  - `electron/mcp-cli.cjs` (plain Node, no Electron import) — use this from WSL/Linux/CI.
- **Storage path**: `$HOME/Documents/SciPaperTodo/database.json` (Linux/WSL respects `HOME`; Windows reads `USERPROFILE`). Both processes can target the same file thanks to the new sentinel-file lock.

## What is intentionally NOT exposed as a tool

- `llm:*` provider mgmt + `llm:startChat` / `llm:approve` — recursive (AI configuring itself / bypassing its own gate)
- `app:bootstrap` / `app:getMcpInfo` — internal init, not user actions
- `block:openAsset` / `block:getPreview` / `article:openFolder` — OS file viewer; meaningless from AI chat
- `block:importAsset` (no path version) — needs file dialog; AI uses `attach_file` with `sourcePath` instead

If you need to expose any of these, document the reason — most are skipped on purpose.

3. **Auto-approve UX**: when `autoApproveTools=true`, the renderer gets a new `'autoApproved'` tool event but no UI listener handles it — silently ignored is fine (tool just runs), but if you want a subtle "auto-approved X" toast, hook into `llmOnEvent` for kind=`autoApproved` in `App.tsx`.
4. **Lock granularity**: current lock is per-write. Many storage funcs do read-modify-write internally; if two writers arrive milliseconds apart, the lock prevents data loss on the OS-level rename, but the second writer's "current state" snapshot may be stale (its merge sees data before the first writer landed). For higher fidelity, switch to read+write under the same held lock. Acceptable for current single-user GUI+MCP scenario.
