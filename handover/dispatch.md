# 子代理派单

## 可调度的子代理

| 名字 | 用法 |
|---|---|
| **codex** (`codex:codex-rescue`) | 重的、跨文件、需要架构判断的活 |
| **kimi** (`kimi:kimi-rescue`) | 独立组件、bounded、清晰输入输出。kimi 可能 unresponsive，重试直至工作 |

## 派单铁律

1. **≤200 行单文件** — 超过拆
2. **输入边界明确** — 列出哪些文件只读，哪些要改
3. **输出契约明确** — TS 严格、不装新依赖、文件路径具体
4. **禁止越界** — "不修改 X、Y、Z"
5. **长输出会断流就拆** — 拆到 ≤150 行，prompt 里禁止它 "snip context"

## Wave 模式

并行起飞：一条消息里多个 Agent tool call → 并行。

实测：6 个并行验过（W7+W9+W2+W10+T1+T2），~25 分钟墙钟。
派 codex 多个并行也 OK（W3+W4 同时跑过）。

## Prompt 模板（实测有效）

```
任务：写 /path/to/file (~XXX 行)。

读这些了解上下文（只读不改）：
- /path/A
- /path/B

要做的：
[具体要求，逐条列]

约束：
- 不修改 X
- 不装新依赖
- TS 严格

输出：
直接 Edit 工具修改 / Write 新文件，不要 markdown 包裹，不要解释。
```

## 断流处理

子代理报 `missing EndStreamResponse` 或在中间停止：

1. 拆任务到 ≤150 行单文件
2. prompt 里加：**"禁止 snip context，禁止省略，逐行写完所有代码"**
3. 如果还是断，让 kimi 接力 codex 没写完的部分

## 什么时候不派子代理

- 简单的 single-file edit（< 50 行）— 主控自己改更快
- 需要看现有代码风格才能做的活 — 主控读完上下文再决定
- 需要跑构建验证的 — 主控跑，子代理跑了你也得自己 verify
- 用户在等回复的活 — 别等子代理，自己上
